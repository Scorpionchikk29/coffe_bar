## ТЕХНИЧЕСКОЕ ЗАДАНИЕ## на разработку веб-приложения «Автоматизированная система заказа кофейни»

---

## 1. ОБЩИЕ СВЕДЕНИЯ## 1.1. Наименование проекта

Веб-приложение «Кофейня» (цифровая витрина, система онлайн-заказа и панель администратора).

## 1.2. Цель разработки

## Создание демонстрационного full-stack коммерческого проекта для портфолио, демонстрирующего навыки проектирования реляционных баз данных, построения REST API и создания адаптивных пользовательских интерфейсов.

## 2. НАЗНАЧЕНИЕ И ЦЕЛИ СОЗДАНИЯ СИСТЕМЫ## 2.1. Назначение системы

Система предназначена для автоматизации процесса заказа позиций меню кофейни (напитки, выпечка) клиентами на условиях самовывоза, а также для оперативного управления этими заказами и меню со стороны персонала (бариста, администратор).

## 2.2. Цели проекта

- Сокращение времени ожидания клиента в очереди за счет предварительного заказа.
- Обеспечение администратора удобным инструментом контроля очереди заказов в реальном времени.
- Сбор базовой статистики продаж для владельца бизнеса.

---

## 3. СТЕК ТЕХНОЛОГИЙ И АРХИТЕКТУРА

- Архитектурный стиль: Клиент-серверная архитектура (микросервисная или монолитная с четким разделением на Frontend и Backend).
- Стек PERN:
- P (PostgreSQL): Реляционная система управления базами данных.
  - E (Express.js): Микрофреймворк для реализации серверной логики.
  - R (React.js): Библиотека для построения пользовательского интерфейса (с использованием TypeScript).
  - N (Node.js): Среда выполнения серверного кода.
- Дополнительно: Prisma ORM для абстракции работы с базой данных, Tailwind CSS для верстки интерфейса.

---

## 4. ТРЕБОВАНИЯ К ФУНКЦИОНАЛЬНОСТИ

Система должна поддерживать три ролевые модели: Гость, Авторизованный клиент и Администратор.

## 4.1. Функциональные требования к модулю «Клиент / Гость»## 4.1.1. Модуль каталога и меню

- Отображение списка доступных товаров, сгруппированных по категориям (Кофе, Чай, Выпечка и др.).
- Фильтрация позиций меню по категориям в один клик без перезагрузки страницы.
- Отображение карточки товара: наименование, описание, базовая цена, изображение, статус наличия.

## 4.1.2. Модуль кастомизации (конструктор продукта)

- Возможность выбора объема напитка (0.2 л, 0.3 л, 0.4 л) с динамическим пересчетом стоимости карточки.
- Возможность выбора добавок: тип молока (кокосовое, соевое, коровье), сиропы (карамель, ваниль и т.д.).

## 4.1.3. Модуль корзины

- Добавление позиций с учетом выбранных параметров кастомизации (одинаковые товары с разным молоком считаются разными позициями корзины).
- Изменение количества единиц товара (+/-) и полное удаление позиции.
- Автоматический расчет промежуточной и итоговой стоимости корзины.

## 4.1.4. Модуль оформления заказа

- Ввод данных покупателя: Имя, Номер телефона.
- Выбор желаемого времени самовывоза (валидация: время не может быть в прошлом).
- Возможность отправки заказа без регистрации (в роли Гостя).

## 4.1.5. Личный кабинет (только для Авторизованных клиентов)

- Просмотр истории прошлых заказов (дата, состав, итоговая стоимость).
- Отслеживание статуса текущего (активного) заказа в режиме реального времени.

---

## 4.2. Функциональные требования к модулю «Администратор»## 4.2.1. Панель мониторинга заказов (Dashboard)

- Отображение единой ленты/канбан-доски со всеми активными заказами.
- Сортировка заказов по времени самовывоза.
- Функционал изменения статуса заказа по цепочке: Принят → Готовится → Готов → Выдан (или Отменен).

## 4.2.2. Модуль управления контентом (CRUD меню)

- Создание новой позиции меню: загрузка фото, ввод названия, описания, цены, привязка к категории.
- Редактирование существующих товаров.
- Управление доступностью: переключатель «в наличии / нет в наличии» (скрывает товар из каталога клиентов).

## 4.2.3. Модуль аналитики и отчетности

- Отображение графика финансовой выручки (за текущий день / текущую неделю).
- Вывод числовых индикаторов: общее количество заказов, средний чек.

---

## 5. СТРУКТУРА ДАННЫХ И СХЕМА БД

База данных должна содержать следующие сущности и связи:

1. Users (Пользователи): Хранение учетных данных.

- Поля: id (PK), email, password_hash, role (client/admin), phone, created_at.

2. Products (Товары): Номенклатура кофейни.

- Поля: id (PK), name, description, price, category_id (FK), image_url, is_available.

3. Categories (Категории): Справочник категорий меню.

- Поля: id (PK), name.

4. Orders (Заказы): Заголовок заказа.

- Поля: id (PK), user_id (FK, nullable для гостей), total_price, status, pickup_time, created_at.

5. Order_Items (Содержимое заказа): Таблица связи многие-ко-многим для заказов и товаров.

- Поля: id (PK), order_id (FK), product_id (FK), quantity, price, customizations (JSON-поле для хранения конфигурации молока/сиропов).

## 6. ТРЕБОВАНИЯ К ИНТЕРФЕЙСУ И API (СПЕЦИФИКАЦИЯ)## 6.1. REST API Эндпоинты

Серверная часть должна строго предоставлять следующий интерфейс:

- Аутентификация:
- POST /api/auth/register — регистрация.
  - POST /api/auth/login — авторизация (выдача токена).
- Управление меню:
- GET /api/products — получение меню потребителем (с фильтрацией).
  - POST /api/products — создание товара (доступ: только Admin).
  - PUT /api/products/:id — обновление товара (доступ: только Admin).
  - DELETE /api/products/:id — удаление товара (доступ: только Admin).
- Управление заказами:
- POST /api/orders — создание нового заказа (публичный эндпоинт).
  - GET /api/orders — список заказов (клиенту — его личные, админу — все).
  - PATCH /api/orders/:id/status — смена статуса заказа (доступ: только Admin).

## 6.2. Требования к UI/UX

- Адаптивность: Интерфейс клиента должен быть спроектирован по принципу Mobile-First (оптимизация под экраны смартфонов). Панель администратора оптимизируется под Desktop/Tablet.
- Валидация: Все формы (ввод телефона, цены, времени) должны проходить валидацию как на клиенте (UI), так и на сервере (API).
